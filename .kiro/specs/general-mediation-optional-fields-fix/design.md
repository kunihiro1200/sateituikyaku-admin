# 一般媒介の任意項目化バグ修正 Design

## Overview

売主リスト通話モードページにおいて、状況（当社）が「一般媒介」の場合に「競合（複数選択可）」と「専任・他決要因（複数選択可）」が誤って必須項目として扱われているバグを修正します。

このバグは、`requiresDecisionDate()` 関数が「一般媒介」を含む全てのケースで `true` を返すため、条件付き必須ロジックが正しく機能していないことが原因です。修正により、一般媒介の売主に対しては両フィールドを任意項目として扱い、専任・他決関連のステータスのみ必須項目として扱います。

## Glossary

- **Bug_Condition (C)**: 状況（当社）が「一般媒介」の場合に発生するバグ条件
- **Property (P)**: 一般媒介の場合は「競合（複数選択可）」と「専任・他決要因（複数選択可）」が任意項目として扱われ、空欄のまま保存できる
- **Preservation**: 専任・他決関連のステータスでは両フィールドが必須項目として扱われ続ける
- **requiresDecisionDate()**: `CallModePage.tsx` の関数（行3068-3072）で、ステータスに基づいて決定日関連フィールドが必須かどうかを判定する
- **isRequiredFieldsComplete()**: `CallModePage.tsx` の関数（行3075-3084）で、必須項目が全て入力されているかをチェックする
- **専任・他決関連ステータス**: 「専任媒介」「他決→専任」「リースバック（専任）」「他決→追客」「他決→追客不要」「一般→他決」「他社買取」

## Bug Details

### Bug Condition

バグは、状況（当社）が「一般媒介」の場合に発生します。`requiresDecisionDate()` 関数が「一般媒介」を含む条件で `true` を返すため、本来任意項目であるべき「競合（複数選択可）」と「専任・他決要因（複数選択可）」が必須項目として扱われます。

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type SellerStatus
  OUTPUT: boolean
  
  RETURN input.status = "一般媒介"
END FUNCTION
```

### Examples

- **例1**: 状況（当社）= "一般媒介"、競合 = []、専任・他決要因 = []
  - **現在の動作（バグ）**: 赤枠と「必須項目です」が表示され、保存時にバリデーションエラー
  - **期待される動作**: 任意項目として表示され、空欄のまま保存可能

- **例2**: 状況（当社）= "専任媒介"、競合 = []、専任・他決要因 = []
  - **現在の動作**: 赤枠と「必須項目です」が表示され、保存時にバリデーションエラー
  - **期待される動作**: 同じ（必須項目として扱われ続ける）

- **例3**: 状況（当社）= "他決→追客"、競合 = []、専任・他決要因 = []
  - **現在の動作**: 赤枠と「必須項目です」が表示され、保存時にバリデーションエラー
  - **期待される動作**: 同じ（必須項目として扱われ続ける）

- **エッジケース**: 状況（当社）= "追客中"
  - **期待される動作**: 「競合（複数選択可）」と「専任・他決要因（複数選択可）」は表示されない（現在の動作を維持）

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- 専任・他決関連のステータス（「専任媒介」「他決→専任」「リースバック（専任）」「他決→追客」「他決→追客不要」「一般→他決」「他社買取」）では、「競合（複数選択可）」と「専任・他決要因（複数選択可）」が必須項目として扱われ続ける
- 「追客中」などの他のステータスでは、両フィールドが表示されない（現在の動作を維持）
- フィールドの表示・非表示ロジックは変更されない

**Scope:**
状況（当社）が「一般媒介」以外の全ての入力は、完全に影響を受けません。これには以下が含まれます：
- 専任・他決関連のステータスでの必須項目チェック
- 「追客中」などでのフィールド非表示
- その他の全てのバリデーションロジック

## Hypothesized Root Cause

バグ説明に基づくと、最も可能性の高い問題は以下の通りです：

1. **不正確な条件判定**: `requiresDecisionDate()` 関数が「一般媒介」を含む条件で `true` を返している
   - 現在の実装: `label.includes('専任') || label.includes('他決') || label.includes('一般媒介')`
   - 「一般媒介」は専任・他決関連ではないため、除外すべき

2. **条件付き必須ロジックの誤り**: `isRequiredFieldsComplete()` 関数が `requiresDecisionDate()` の結果に依存しているため、一般媒介でも必須チェックが実行される

3. **UIレンダリングロジックの連動**: フィールドの `required` プロパティと `error` プロパティが `requiresDecisionDate()` の結果に基づいて設定されている

## Correctness Properties

Property 1: Bug Condition - 一般媒介の任意項目化

_For any_ 状況（当社）が「一般媒介」の入力に対して、修正後の `requiresDecisionDate` 関数は `false` を返し、「競合（複数選択可）」と「専任・他決要因（複数選択可）」が任意項目として扱われ、空欄のまま保存できる。

**Validates: Requirements 2.1, 2.2, 2.3, 2.4**

Property 2: Preservation - 専任・他決関連の必須項目維持

_For any_ 状況（当社）が「一般媒介」以外の入力（特に専任・他決関連のステータス）に対して、修正後の関数は修正前と同じ結果を返し、「競合（複数選択可）」と「専任・他決要因（複数選択可）」が必須項目として扱われ続ける。

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5**

## Fix Implementation

### Changes Required

根本原因分析が正しいと仮定すると：

**File**: `frontend/frontend/src/pages/CallModePage.tsx`

**Function**: `requiresDecisionDate()` （行3068-3072）

**Specific Changes**:
1. **条件判定の修正**: 「一般媒介」を除外し、専任・他決関連のステータスのみを対象とする
   - 修正前: `return label.includes('専任') || label.includes('他決') || label.includes('一般媒介');`
   - 修正後: `return label.includes('専任') || label.includes('他決');`

2. **バリデーションロジックの確認**: `isRequiredFieldsComplete()` 関数が正しく動作することを確認
   - `requiresDecisionDate()` が `false` を返す場合、必須チェックがスキップされることを確認

3. **UIレンダリングの確認**: フィールドの `required` と `error` プロパティが正しく設定されることを確認
   - 一般媒介の場合、赤枠と「必須項目です」メッセージが表示されないことを確認

## Testing Strategy

### Validation Approach

テスト戦略は2段階のアプローチに従います：まず、修正前のコードでバグを実証する反例を表面化させ、次に修正が正しく機能し、既存の動作を保持することを検証します。

### Exploratory Bug Condition Checking

**Goal**: 修正を実装する前に、バグを実証する反例を表面化させる。根本原因分析を確認または反証する。反証した場合、再仮説が必要。

**Test Plan**: 状況（当社）を「一般媒介」に設定し、「競合（複数選択可）」と「専任・他決要因（複数選択可）」を空欄のまま保存しようとするテストを作成する。修正前のコードで実行し、失敗を観察して根本原因を理解する。

**Test Cases**:
1. **一般媒介の必須チェックテスト**: 状況（当社）= "一般媒介"、競合 = []、専任・他決要因 = [] で保存を試みる（修正前のコードで失敗）
2. **専任媒介の必須チェックテスト**: 状況（当社）= "専任媒介"、競合 = []、専任・他決要因 = [] で保存を試みる（修正前のコードで失敗、修正後も失敗すべき）
3. **他決→追客の必須チェックテスト**: 状況（当社）= "他決→追客"、競合 = []、専任・他決要因 = [] で保存を試みる（修正前のコードで失敗、修正後も失敗すべき）
4. **追客中の非表示テスト**: 状況（当社）= "追客中" で両フィールドが表示されないことを確認（修正前後で変わらない）

**Expected Counterexamples**:
- 一般媒介の場合、空欄で保存しようとするとバリデーションエラーが発生
- 可能性のある原因: `requiresDecisionDate()` が「一般媒介」を含む条件で `true` を返している、UIレンダリングロジックが連動している

### Fix Checking

**Goal**: バグ条件が成立する全ての入力に対して、修正後の関数が期待される動作を生成することを検証する。

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := requiresDecisionDate_fixed(input.status)
  ASSERT result = false
  ASSERT canSaveWithEmptyFields(input) = true
END FOR
```

### Preservation Checking

**Goal**: バグ条件が成立しない全ての入力に対して、修正後の関数が修正前の関数と同じ結果を生成することを検証する。

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT requiresDecisionDate_original(input.status) = requiresDecisionDate_fixed(input.status)
END FOR
```

**Testing Approach**: プロパティベーステストは、保存チェックに推奨されます。理由は以下の通りです：
- 入力ドメイン全体で多くのテストケースを自動的に生成する
- 手動ユニットテストが見逃す可能性のあるエッジケースをキャッチする
- バグのない入力に対して動作が変更されていないという強力な保証を提供する

**Test Plan**: 修正前のコードで専任・他決関連のステータスの動作を観察し、その動作をキャプチャするプロパティベーステストを作成する。

**Test Cases**:
1. **専任媒介の保存チェック**: 状況（当社）= "専任媒介" で、修正前後で必須チェックが同じことを確認
2. **他決→専任の保存チェック**: 状況（当社）= "他決→専任" で、修正前後で必須チェックが同じことを確認
3. **リースバック（専任）の保存チェック**: 状況（当社）= "リースバック（専任）" で、修正前後で必須チェックが同じことを確認
4. **他決→追客の保存チェック**: 状況（当社）= "他決→追客" で、修正前後で必須チェックが同じことを確認

### Unit Tests

- 状況（当社）が「一般媒介」の場合、`requiresDecisionDate()` が `false` を返すことをテスト
- 状況（当社）が専任・他決関連の場合、`requiresDecisionDate()` が `true` を返すことをテスト
- エッジケース（空文字列、null、未定義）をテスト

### Property-Based Tests

- ランダムな状況（当社）を生成し、一般媒介の場合は任意項目、専任・他決関連の場合は必須項目として扱われることを検証
- ランダムな入力データを生成し、修正前後で専任・他決関連のステータスの動作が同じことを検証
- 多くのシナリオで全ての非一般媒介入力が引き続き正しく動作することをテスト

### Integration Tests

- 通話モードページで状況（当社）を「一般媒介」に変更し、両フィールドが任意項目として表示されることをテスト
- 通話モードページで状況（当社）を「専任媒介」に変更し、両フィールドが必須項目として表示されることをテスト
- 通話モードページで状況（当社）を「一般媒介」に設定し、空欄のまま保存できることをテスト
