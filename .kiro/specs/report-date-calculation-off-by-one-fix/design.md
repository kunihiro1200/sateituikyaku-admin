# 報告日計算の1日ずれバグ修正設計

## Overview

物件リストスプレッドシートの「報告日」フィールドで、「報告日設定」を「する」に設定した際に報告日が本日+13日で計算されているが、正しくは本日+14日であるべき。本設計書では、このoff-by-oneエラーの根本原因を特定し、修正方法を定義する。

## Glossary

- **Bug_Condition (C)**: 「報告日設定」が「する」で、計算結果が本日+13日になる条件
- **Property (P)**: 報告日が本日+14日で正確に計算される動作
- **Preservation**: 「報告日設定」が「しない」の場合や手動入力された報告日は変更されない
- **報告日設定**: 物件リストスプレッドシートの列で、「する」または「しない」を選択
- **報告日**: 物件リストスプレッドシートの列で、日付が自動計算または手動入力される
- **TODAY()**: スプレッドシートの関数で、現在の日付を返す

## Bug Details

### Bug Condition

報告日の自動計算が本日+13日になっているが、正しくは本日+14日であるべき。

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type PropertyReportDateInput
  OUTPUT: boolean
  
  RETURN input.reportDateSetting = "する" AND
         input.calculatedReportDate = TODAY() + 13 AND
         NOT (input.calculatedReportDate = TODAY() + 14)
END FUNCTION
```

### Examples

- **物件番号AA12497**: 本日が2026年3月31日の場合
  - 現在の計算結果: 2026年4月13日（本日+13日）← バグ
  - 期待される結果: 2026年4月14日（本日+14日）← 正しい

- **一般的なケース**: 本日が2026年1月1日の場合
  - 現在の計算結果: 2026年1月14日（本日+13日）← バグ
  - 期待される結果: 2026年1月15日（本日+14日）← 正しい

- **月をまたぐケース**: 本日が2026年2月20日の場合
  - 現在の計算結果: 2026年3月5日（本日+13日）← バグ
  - 期待される結果: 2026年3月6日（本日+14日）← 正しい

- **うるう年のケース**: 本日が2024年2月20日の場合
  - 現在の計算結果: 2024年3月4日（本日+13日）← バグ
  - 期待される結果: 2024年3月5日（本日+14日）← 正しい

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- 「報告日設定」が「しない」の場合、報告日は自動計算されず、手動入力が継続して受け付けられる
- 報告日が手動で入力された場合、その値は保持される
- 「報告日設定」以外のフィールドを操作した場合、報告日の値は変更されない

**Scope:**
「報告日設定」が「する」以外の全ての入力は、完全に影響を受けない。これには以下が含まれる：
- 「報告日設定」が「しない」の場合
- 「報告日設定」が空欄の場合
- 報告日が手動で入力されている場合

## Hypothesized Root Cause

バグ修正要件の分析に基づき、根本原因が確定しました：

**確定した根本原因: Glideアプリの「報告日」カラム設定が間違っている**

**場所**: Glideアプリの「報告日」カラム設定

**現在の設定**:
- **Reset on edit**: `[報告日設定] <> [_THISROW_BEFORE].[報告日設定], [報告完了] <> [_THISROW_BEFORE].[報告完了]`
- **Initial value**: `IF(ISNOTBLANK([_THISROW_BEFORE].[報告日]), [_THISROW_BEFORE].[報告日] + 14, TODAY() + 14)`

**問題点**:
- `[_THISROW_BEFORE].[報告日] + 14` は**前回の報告日に14日を足す**計算になっている
- AA12497の場合、本来は空欄だったが、何らかの理由で古い値（3月31日）が入っていた
- その結果、3月31日 + 14日 = 4月14日ではなく、3月31日 - 1日 + 14日 = 4月13日になった（推測）
- **重要な事実**: 前回の報告日は存在しない（本来は空欄）
- 「報告日設定」で「する」を選択した時点で本日+14日（4月14日）に設定されるべきだった

**修正方法**:
Glideアプリの「報告日」カラムのInitial valueを以下に変更:
```
TODAY() + 14
```

これで常に本日+14日が設定されます。前回の報告日を参照しないため、古い値の影響を受けません。

## Correctness Properties

Property 1: Bug Condition - 報告日が本日+14日で正確に計算される

_For any_ 物件で「報告日設定」が「する」に設定された場合、修正後のシステムは報告日を本日+14日で正確に計算し、本日+13日にはならない。

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - 手動入力と「しない」設定の動作が保持される

_For any_ 物件で「報告日設定」が「しない」または空欄の場合、または報告日が手動で入力されている場合、修正後のシステムは修正前と全く同じ動作を保持し、報告日の値を変更しない。

**Validates: Requirements 3.1, 3.2, 3.3**

## Fix Implementation

### Changes Required

根本原因が確定したため、以下の修正が必要：

**修正箇所**: Glideアプリの「報告日」カラム設定

**Specific Changes**:
1. **Glideアプリを開く**:
   - Glideアプリの編集画面を開く
   - 「報告日」カラムを選択

2. **Initial valueを修正**:
   - 現在の設定: `IF(ISNOTBLANK([_THISROW_BEFORE].[報告日]), [_THISROW_BEFORE].[報告日] + 14, TODAY() + 14)`
   - 修正後の設定: `TODAY() + 14`
   - 保存

3. **AA12497の報告日を修正**:
   - 現在の間違った値（4月13日）を削除または正しい値（4月14日）に修正
   - または、「報告日設定」を一度「しない」に変更してから「する」に戻して再計算させる

4. **動作確認**:
   - AA12497で「報告日設定」を「する」に設定
   - 報告日が本日+14日（4月14日）になることを確認

**注意**: フロントエンド（`ReinsRegistrationPage.tsx`）に実装した報告日自動計算ロジックは、Glideアプリの設定が修正されるまでの一時的な対策として機能します。Glideアプリの設定が修正されれば、フロントエンドのロジックは不要になる可能性がありますが、二重の保護として残しておくことも検討できます。

## Testing Strategy

### Validation Approach

テスト戦略は2段階のアプローチに従う：まず、修正前のコードでバグを再現し、次に修正後のコードで正しい動作を検証する。

### Exploratory Bug Condition Checking

**Goal**: 修正前にバグを再現し、根本原因を確認する。根本原因の仮説が正しいか検証する。

**Test Plan**: 物件リストスプレッドシートで「報告日設定」を「する」に設定し、報告日が本日+13日になることを確認する。修正前のコードでテストを実行し、失敗を観察して根本原因を理解する。

**Test Cases**:
1. **AA12497のテスト**: 物件番号AA12497で「報告日設定」を「する」に設定し、報告日が本日+13日になることを確認（修正前は失敗）
2. **新規物件のテスト**: 新しい物件を作成し、「報告日設定」を「する」に設定し、報告日が本日+13日になることを確認（修正前は失敗）
3. **月をまたぐケース**: 本日が月末の場合、報告日が翌月の正しい日付になるか確認（修正前は1日ずれる）
4. **うるう年のケース**: 本日が2月の場合、報告日がうるう年を考慮して計算されるか確認（修正前は1日ずれる）

**Expected Counterexamples**:
- 報告日が本日+13日になる（本来は+14日であるべき）
- 可能性のある原因: スプレッドシートの数式が `+13` になっている、GASの計算ロジックが `+13` を使用している

### Fix Checking

**Goal**: 修正後のコードで、「報告日設定」が「する」の場合に報告日が本日+14日で正確に計算されることを検証する。

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := calculateReportDate_fixed(input)
  ASSERT result = TODAY() + 14
  ASSERT NOT (result = TODAY() + 13)
END FOR
```

### Preservation Checking

**Goal**: 修正後のコードで、「報告日設定」が「しない」または空欄の場合、または報告日が手動で入力されている場合、修正前と全く同じ動作を保持することを検証する。

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT calculateReportDate_original(input) = calculateReportDate_fixed(input)
END FOR
```

**Testing Approach**: Property-based testingは、保存チェックに推奨される。なぜなら：
- 入力ドメイン全体で多くのテストケースを自動的に生成する
- 手動の単体テストでは見逃す可能性のあるエッジケースをキャッチする
- 非バグ入力に対して動作が変更されていないことを強力に保証する

**Test Plan**: 修正前のコードで「報告日設定」が「しない」の場合や手動入力の動作を観察し、その動作をキャプチャするproperty-based testを書く。

**Test Cases**:
1. **「しない」設定の保持**: 「報告日設定」が「しない」の物件で、報告日が自動計算されないことを確認
2. **手動入力の保持**: 報告日が手動で入力されている物件で、その値が保持されることを確認
3. **空欄の保持**: 「報告日設定」が空欄の物件で、報告日が自動計算されないことを確認
4. **他のフィールドの保持**: 「報告日設定」以外のフィールドを変更した場合、報告日が変更されないことを確認

### Unit Tests

- 「報告日設定」が「する」の場合、報告日が本日+14日になることをテスト
- 「報告日設定」が「しない」の場合、報告日が自動計算されないことをテスト

- 多くのシナリオで、手動入力された報告日が保持されることをテスト

### Integration Tests

- 物件リストスプレッドシートで「報告日設定」を「する」に設定し、データベースに同期された報告日が本日+14日になることを確認
