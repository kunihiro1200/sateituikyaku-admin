# バグ修正要件ドキュメント

## はじめに

共有ページ（`/shared-items`）で新規作成したエントリー（例: ID: 212）を保存した直後は表示されるが、しばらく後に消えてしまうバグを修正します。

システムの構造上、`SharedItemsService` はスプレッドシートのみをデータストアとして使用しています（DBは使用していません）。そのため「DBから消えた」という現象は「スプレッドシートから行が消えた」ことを意味します。

ユーザーの推測に基づくと、以下のシナリオが考えられます：
1. 保存時にスプレッドシートへの `appendRow` が失敗またはタイムアウトした
2. その後、GASによる定期同期（スプシ→スプシ）が実行された際に、何らかの理由で新規行が上書き・削除された

---

## バグ分析

### 現在の動作（不具合）

1.1 WHEN ユーザーが共有ページで新規エントリーを保存したとき THEN システムは一時的にエントリーを表示するが、しばらく後にそのエントリーが消える

1.2 WHEN スプレッドシートへの `appendRow` がタイムアウトまたはエラーで失敗したとき THEN システムはエラーを表示せずに保存成功として扱い、エントリーがスプレッドシートに存在しない状態になる

1.3 WHEN GASによる定期同期（スプシ→スプシ）が実行されたとき THEN システムはスプレッドシートに存在しない新規エントリーをDBキャッシュや表示から削除する

### 期待される動作（正しい動作）

2.1 WHEN ユーザーが共有ページで新規エントリーを保存したとき THEN システムはスプレッドシートへの保存が完全に成功したことを確認してから保存完了とし、エントリーが永続的に保持される

2.2 WHEN スプレッドシートへの `appendRow` が失敗したとき THEN システムはエラーメッセージを表示してフォームを閉じず、ユーザーに再試行を促す

2.3 WHEN 保存後にスプレッドシートを読み直したとき THEN システムは保存したエントリーがスプレッドシートに実際に存在することを確認できる

### 変わらない動作（リグレッション防止）

3.1 WHEN スプレッドシートへの保存が正常に成功したとき THEN システムは引き続き保存完了後にフォームを閉じてエントリー一覧を更新する

3.2 WHEN 既存のエントリーが正常に存在するとき THEN システムは引き続き既存エントリーの一覧表示・詳細表示を正常に行う

3.3 WHEN ユーザーが保存ボタンをクリックしたとき THEN システムは引き続き全ての必須フィールドのバリデーションを実行する

3.4 WHEN 保存処理が実行中のとき THEN システムは引き続き保存ボタンを非活性化して二重送信を防止する
