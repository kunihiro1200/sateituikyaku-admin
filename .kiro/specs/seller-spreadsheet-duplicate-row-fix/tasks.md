# 実装計画

- [x] 1. バグ条件探索テストを作成
  - **Property 1: Bug Condition** - 既存行検索の失敗（競合状態）
  - **重要**: このテストは未修正コードで実行し、失敗することを確認する（失敗がバグの存在を証明）
  - **目的**: 反例を表面化し、バグが存在することを確認する
  - **スコープ付きPBTアプローチ**: 決定論的バグのため、具体的な失敗ケースにプロパティをスコープする
  - 同じ売主番号（例: AA13888）に対して複数の同期リクエストを短時間（0.5秒間隔）で実行するテストを作成
  - バグ条件: `isBugCondition(syncRequest)` - 既存行が存在するが`findRowByColumn()`が`null`を返し、競合状態または
キャッシュ無効化が発生している
  - 期待される動作: 既存行が見つかり更新される（設計書のExpected Behavior Propertiesより）
  - 未修正コードで実行
  - **期待される結果**: テストが失敗する（これは正しい - バグが存在することを証明）
  - 反例を文書化してルート原因を理解する
  - テストが作成され、実行され、失敗が文書化されたらタスク完了とする
  - _Requirements: 1.1, 1.2, 1.3, 2.1, 2.2, 2.3_

- [x] 2. 保存プロパティテストを作成（修正実装前）
  - **Property 2: Preservation** - 新規行追加と通常の同期の正確性
  - **重要**: 観察優先の方法論に従う
  - 未修正コードでバグのない入力（新規売主登録、競合のない既存売主更新）の動作を観察
  - 観察された動作パターンをキャプチャするプロパティベーステストを作成（設計書のPreservation Requirementsより）
  - プロパティベーステストは多くのテストケースを生成し、より強力な保証を提供
  - 未修正コードでテストを実行
  - **期待される結果**: テストが成功する（ベースライン動作を確認）
  - テストが作成され、実行され、未修正コードで成功したらタスク完了とする
  - _Requirements: 3.1, 3.2, 3.3, 3.4_

  - [x] 3.1 修正を実装

    - `GoogleSheetsClient.findRowByColumn()`の検索範囲を明示的に指定
    - Google Sheets APIエラー時のリトライを追加
    - _Bug_Condition: isBugCondition(syncRequest) - 既存行が存在するが`findRowByColumn()`が`null`を返し、複数の同期リクエストが同時実行されるか、キャッシュ無効化が発生_
