# 実装計画

- [ ] 1. バグ条件探索テストを作成（修正前）
  - **Property 1: Bug Condition** - サイドバーカウントエンドポイントの401エラー
  - **重要**: このテストは未修正のコードで実行し、**失敗することを確認する**（失敗がバグの存在を証明する）
  - **修正しようとしない**: テストが失敗しても、テストやコードを修正しない
  - **注意**: このテストは期待される動作をエンコードしており、修正後にパスすることで修正を検証する
  - **目標**: バグの存在を示す具体例を明らかにする
  - **スコープ付きPBTアプローチ**: 決定的なバグの場合、プロパティを具体的な失敗ケースにスコープして再現性を確保する
  - テスト実装の詳細（design.mdのBug Conditionセクションより）:
    - `/api/sellers/sidebar-counts`エンドポイントにGETリクエストを送信
    - 認証トークンなしでリクエスト
    - レスポンスステータスが401であることを確認（未修正コードでの期待される動作）
  - テストアサーションはdesign.mdのExpected Behavior Propertiesと一致させる:
    - 修正後は200 OKレスポンスを返すべき
    - 修正後は全カテゴリ（専任、一般、未訪問他決、訪問後他決）のカウントを含むべき
  - 未修正のコードでテストを実行
  - **期待される結果**: テストが失敗する（これは正しい - バグの存在を証明する）
  - 失敗例を記録して根本原因を理解する
  - テストが作成され、実行され、失敗が記録されたらタスク完了とする
  - _Requirements: 1.1, 1.2, 1.3, 1.4_

- [ ] 2. 保存プロパティテストを作成（修正前）
  - **Property 2: Preservation** - 他のエンドポイントの認証動作
  - **重要**: 観察優先の方法論に従う
  - 未修正のコードで非バグ入力（認証が必要なエンドポイント）の動作を観察する
  - Preservation Requirementsから観察された動作パターンをキャプチャするプロパティベーステストを作成する:
    - 認証が必要なエンドポイント（`POST /api/sellers`, `PUT /api/sellers/:id`など）は認証なしで401エラーを返す
    - 他の認証不要のエンドポイント（`/api/buyers/sidebar-counts`など）は認証なしで200 OKを返す
    - サイドバーカウントロジック（`getSidebarCountsFallback()`）は同じ結果を返す
  - プロパティベーステストは多くのテストケースを生成し、より強力な保証を提供する
  - 未修正のコードでテストを実行
  - **期待される結果**: テストがパスする（保存すべきベースライン動作を確認する）
  - テストが作成され、実行され、未修正のコードでパスしたらタスク完了とする
  - _Requirements: 3.1, 3.2, 3.3, 3.4_
