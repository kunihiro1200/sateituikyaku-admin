# 売主番号プレフィックス検索対応 バグ修正設計書

## Overview

売主リストの検索機能において、売主番号のプレフィックスが`AA`のみに限定されている問題を修正します。現在の正規表現`/^aa\d+$/i`を`/^[a-z]{2}\d+$/i`に変更することで、`FI`、`BB`、`CC`等の任意の2文字アルファベットプレフィックスに対応します。

この修正により、`AA`以外のプレフィックスを持つ売主番号（例: `FI123`、`BB456`、`CC789`）も正常に検索できるようになります。

## Glossary

- **Bug_Condition (C)**: 売主番号検索で`AA`以外の2文字アルファベットプレフィックス + 数字の組み合わせを入力した場合
- **Property (P)**: 正規表現にマッチし、該当する売主を検索結果として返す
- **Preservation**: `AA`プレフィックスの検索、無効な形式の検索、売主番号以外の検索が正常に動作し続けること
- **searchSellers**: `backend/src/services/SellerService.supabase.ts`の1538行目付近にあるメソッド。売主番号での検索を実行する
- **seller_number**: データベースの`sellers`テーブルに格納されている売主番号カラム（例: `AA13501`、`FI00123`）

## Bug Details

### Bug Condition

バグは、ユーザーが売主番号検索バーに`AA`以外の2文字アルファベットプレフィックス + 数字の組み合わせ（例: `FI123`、`BB456`、`CC789`）を入力した場合に発生します。`searchSellers`メソッドは正規表現`/^aa\d+$/i`を使用しているため、`AA`プレフィックスのみにマッチし、他のプレフィックスは検索されません。

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type string (検索クエリ)
  OUTPUT: boolean
  
  RETURN input.match(/^[a-z]{2}\d+$/i) !== null
         AND NOT input.match(/^aa\d+$/i)
         AND sellerWithPrefixExists(input)
END FUNCTION
```

### Examples

- **例1**: ユーザーが`FI123`を入力 → 正規表現にマッチせず、検索が実行されない（期待: `FI00123`の売主が検索結果に表示される）
- **例2**: ユーザーが`BB456`を入力 → 正規表現にマッチせず、検索が実行されない（期待: `BB00456`の売主が検索結果に表示される）
- **例3**: ユーザーが`CC789`を入力 → 正規表現にマッチせず、検索が実行されない（期待: `CC00789`の売主が検索結果に表示される）
- **エッジケース**: ユーザーが`AA123`を入力 → 正規表現にマッチし、`AA00123`の売主が検索結果に表示される（期待通り）

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- `AA`プレフィックスを持つ売主番号の検索は引き続き正常に動作する
- 無効な形式（例: `A123`、`ABC123`、`12345`）の検索は引き続き正規表現にマッチせず、検索が実行されない
- 売主番号以外の検索条件（名前、住所等）での検索は引き続き正常に動作する

**Scope:**
`AA`プレフィックスを持つ売主番号、無効な形式の入力、売主番号以外の検索条件は、この修正の影響を受けません。これには以下が含まれます：
- `AA13501`のような`AA`プレフィックスの売主番号検索
- `A123`、`ABC123`、`12345`のような無効な形式の入力
- 名前、住所、電話番号等の売主番号以外の検索条件

## Hypothesized Root Cause

バグ説明に基づくと、最も可能性の高い原因は以下の通りです：

1. **正規表現の限定**: `searchSellers`メソッドが`/^aa\d+$/i`という正規表現を使用しており、`AA`プレフィックスのみにマッチする
   - 1543行目: `if (lowerQuery.match(/^aa\d+$/i))`
   - この条件により、`AA`以外のプレフィックスは「売主番号での検索」として認識されない

2. **プレフィックス抽出ロジックの固定**: 1548行目で`const numPart = upperQuery.replace(/^AA/i, '');`として`AA`プレフィックスを固定で削除している
   - 他のプレフィックス（`FI`、`BB`、`CC`等）には対応していない

3. **ゼロパディングロジックの固定**: 1549行目で`const paddedQuery = `AA${numPart.padStart(5, '0')}`;`として`AA`プレフィックスを固定で付与している
   - 他のプレフィックスには対応していない

## Fix Implementation

### Changes Required

根本原因分析が正しいと仮定すると：

**File**: `backend/src/services/SellerService.supabase.ts`

**Function**: `searchSellers`（1538行目付近）

**Specific Changes**:
1. **正規表現の変更**: 1543行目の`/^aa\d+$/i`を`/^[a-z]{2}\d+$/i`に変更
   - 変更前: `if (lowerQuery.match(/^aa\d+$/i))`
   - 変更後: `if (lowerQuery.match(/^[a-z]{2}\d+$/i))`

2. **プレフィックス抽出ロジックの汎用化**: 1548行目の`replace(/^AA/i, '')`を`replace(/^[a-z]{2}/i, '')`に変更
   - 変更前: `const numPart = upperQuery.replace(/^AA/i, '');`
   - 変更後: `const numPart = upperQuery.replace(/^[a-z]{2}/i, '');`

3. **プレフィックス取得ロジックの追加**: 1549行目の前に、入力からプレフィックスを抽出するロジックを追加
   - 追加: `const prefix = upperQuery.match(/^[a-z]{2}/i)?.[0] || 'AA';`

4. **ゼロパディングロジックの汎用化**: 1549行目の`` `AA${numPart.padStart(5, '0')}` ``を`` `${prefix}${numPart.padStart(5, '0')}` ``に変更
   - 変更前: `` const paddedQuery = `AA${numPart.padStart(5, '0')}`; ``
   - 変更後: `` const paddedQuery = `${prefix}${numPart.padStart(5, '0')}`; ``

5. **コメントの更新**: 1544-1545行目のコメントを更新
   - 変更前: `// AA12903のような形式の場合、データベースで直接検索`
   - 変更後: `// AA12903、FI123、BB456のような形式の場合、データベースで直接検索`

## Testing Strategy

### Validation Approach

テスト戦略は2段階のアプローチに従います：まず、修正前のコードでバグを実証する反例を表面化させ、次に修正が正しく動作し、既存の動作を保持することを検証します。

### Exploratory Bug Condition Checking

**Goal**: 修正を実装する前に、バグを実証する反例を表面化させる。根本原因分析を確認または反証する。反証した場合は、再仮説が必要です。

**Test Plan**: `FI123`、`BB456`、`CC789`等の`AA`以外のプレフィックスを持つ売主番号で検索を実行するテストを作成します。これらのテストを修正前のコードで実行し、失敗を観察して根本原因を理解します。

**Test Cases**:
1. **FI123検索テスト**: `FI123`を検索（修正前のコードでは失敗）
2. **BB456検索テスト**: `BB456`を検索（修正前のコードでは失敗）
3. **CC789検索テスト**: `CC789`を検索（修正前のコードでは失敗）
4. **無効な形式テスト**: `A123`、`ABC123`、`12345`を検索（修正前のコードでも失敗、期待通り）

**Expected Counterexamples**:
- `FI123`、`BB456`、`CC789`の検索が正規表現にマッチせず、検索が実行されない
- 可能な原因: 正規表現が`AA`プレフィックスのみにマッチ、プレフィックス抽出ロジックが固定

### Fix Checking

**Goal**: バグ条件が成立する全ての入力に対して、修正後の関数が期待される動作を生成することを検証します。

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := searchSellers_fixed(input)
  ASSERT expectedBehavior(result)
END FOR
```

### Preservation Checking

**Goal**: バグ条件が成立しない全ての入力に対して、修正後の関数が元の関数と同じ結果を生成することを検証します。

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT searchSellers_original(input) = searchSellers_fixed(input)
END FOR
```

**Testing Approach**: 保存チェックにはプロパティベーステストが推奨されます。理由は以下の通りです：
- 入力ドメイン全体で多くのテストケースを自動的に生成する
- 手動の単体テストが見逃す可能性のあるエッジケースをキャッチする
- 全ての非バグ入力に対して動作が変更されていないという強力な保証を提供する

**Test Plan**: まず修正前のコードで`AA`プレフィックスの検索、無効な形式の検索、売主番号以外の検索の動作を観察し、その動作をキャプチャするプロパティベーステストを作成します。
