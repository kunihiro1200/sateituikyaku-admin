# 物件リスト報告ページ「未報告」カテゴリー判定バグ修正 設計ドキュメント

## Overview

物件リストの「報告」ページにおいて、報告日が設定されている物件（例：AA12636、報告日4/14）が「未報告」カテゴリーに誤って表示される問題を修正します。

根本原因は `frontend/frontend/src/utils/propertyListingStatusUtils.ts` の104-113行目の判定ロジックが逆になっていることです。現在のロジックは「報告日が設定されていて、かつ今日以前」→「未報告」と判定していますが、正しくは「報告日が未設定、または報告日が未来」→「未報告」と判定すべきです。

この修正により、報告日が設定されている物件は「報告済み」として扱われ、「未報告」カテゴリーから除外されます。

## Glossary

- **Bug_Condition (C)**: 報告日が設定されていて、かつ今日以前である物件が「未報告」カテゴリーに表示される条件
- **Property (P)**: 報告日が未設定、または報告日が未来の物件のみが「未報告」カテゴリーに表示されるべき正しい動作
- **Preservation**: 他のステータス判定ロジック（公開中、非公開など）や報告担当者ラベル表示が変更されないこと
- **calculatePropertyStatus**: `frontend/frontend/src/utils/propertyListingStatusUtils.ts` の関数で、物件のステータスを計算する
- **report_date**: 物件リストテーブルの報告日カラム（DATE型）
- **report_assignee**: 物件リストテーブルの報告担当者カラム（TEXT型）

## Bug Details

### Bug Condition

バグは、報告日が設定されていて、かつ今日以前である物件が「未報告」カテゴリーに表示される場合に発生します。`calculatePropertyStatus` 関数の104-113行目の判定ロジックが逆になっています。

**Formal Specification:**
```
FUNCTION isBugCondition(listing)
  INPUT: listing of type PropertyListing
  OUTPUT: boolean
  
  RETURN listing.report_date IS NOT NULL
         AND parseDate(listing.report_date) <= today
         AND currentLogic(listing) returns 'unreported'
END FUNCTION
```

### Examples

- **AA12636**: 報告日が4/14（今日以前）に設定されている → 現在は「未報告」に表示される（バグ） → 修正後は「未報告」に表示されない（正しい）
- **AA12637**: 報告日が未設定（null） → 現在は「未報告」に表示されない（バグ） → 修正後は「未報告」に表示される（正しい）
- **AA12638**: 報告日が未来の日付（例：明日） → 現在は「未報告」に表示されない（バグ） → 修正後は「未報告」に表示される（正しい）
- **AA12639**: 報告日が今日 → 現在は「未報告」に表示される（バグ） → 修正後は「未報告」に表示されない（正しい）

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- 報告日が今日以前に設定されている物件が「未報告」以外のカテゴリーに表示される場合、その表示を継続する
- 報告担当者（report_assignee）が設定されている場合、「未報告{担当者名}」の形式でラベルを表示し続ける
- 他のステータス判定ロジック（公開中、非公開、値下げ予定など）が動作している場合、それらのロジックを変更せずに動作し続ける

**Scope:**
報告日の判定ロジック以外の全ての動作は完全に変更されません。これには以下が含まれます：
- 値下げ予定日の判定（優先度0）
- 確認ステータスの判定（優先度2）
- 一般媒介非公開の判定（優先度3）
- １社掲載の判定（優先度4）
- 公開予定日の判定（優先度5）
- SUUMO URL登録の判定（優先度6）
- 買付申込みの判定（優先度7）
- 公開前情報の判定（優先度8）
- 非公開（配信メールのみ）の判定（優先度9）
- 一般公開中の判定（優先度10）
- 専任公開中の判定（優先度11）

## Hypothesized Root Cause

バグの根本原因は、`calculatePropertyStatus` 関数の104-113行目の判定ロジックが逆になっていることです。

1. **判定条件の逆転**: 現在のロジックは `if (reportDate && reportDate <= today)` で「報告日が設定されていて、かつ今日以前」→「未報告」と判定しています。しかし、正しくは「報告日が未設定、または報告日が未来」→「未報告」と判定すべきです。

2. **論理演算子の誤り**: 現在のロジックは `&&`（AND）を使用していますが、正しくは `||`（OR）を使用すべきです。または、条件を反転させて `!reportDate || reportDate > today` とすべきです。

3. **ビジネスロジックの誤解**: 「報告日が設定されている」ということは「報告済み」を意味するため、「未報告」カテゴリーから除外すべきです。現在のロジックはこの理解が逆になっています。

## Correctness Properties

Property 1: Bug Condition - 報告日未設定または未来の物件のみ「未報告」に表示

_For any_ 物件リストレコードにおいて、報告日が未設定（null）であるか、または報告日が未来の日付である場合、修正後の calculatePropertyStatus 関数 SHALL その物件を「未報告」カテゴリーに分類し、報告担当者が設定されている場合は「未報告{担当者名}」のラベルを返す。

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - 報告日が今日以前の物件は「未報告」に表示されない

_For any_ 物件リストレコードにおいて、報告日が今日以前に設定されている場合、修正後の calculatePropertyStatus 関数 SHALL その物件を「未報告」カテゴリーに分類せず、他のステータス判定ロジック（優先度2以降）に従って適切なカテゴリーに分類する。

**Validates: Requirements 2.3, 3.1, 3.3**

## Fix Implementation

### Changes Required

根本原因分析が正しいと仮定した場合の修正内容：

**File**: `frontend/frontend/src/utils/propertyListingStatusUtils.ts`

**Function**: `calculatePropertyStatus`

**Specific Changes**:
1. **判定条件の反転**（104-113行目）:
   - 現在: `if (reportDate && reportDate <= today)`
   - 修正後: `if (!reportDate || reportDate > today)`
   - 説明: 「報告日が未設定、または報告日が未来」の場合に「未報告」と判定する

2. **コメントの修正**:
   - 現在: `// 1. 報告日が今日以前で未報告`
   - 修正後: `// 1. 報告日が未設定または未来で未報告`
   - 説明: コメントを正しいロジックに合わせる

3. **ラベル表示の維持**:
   - `report_assignee` が設定されている場合、引き続き「未報告{担当者名}」の形式でラベルを表示する
   - この部分は変更不要

4. **優先度の維持**:
   - 「未報告」判定は優先度1のまま維持（値下げ予定日の次）
   - 他のステータス判定ロジックの優先度は変更しない

## Testing Strategy

### Validation Approach

テスト戦略は2段階アプローチに従います：まず、未修正コードでバグを再現する探索的テストを実行し、次に修正後のコードで正しい動作を検証し、既存の動作が保持されていることを確認します。

### Exploratory Bug Condition Checking

**Goal**: 未修正コードでバグを再現し、根本原因分析を確認または反証する。反証された場合は、根本原因を再分析する必要があります。

**Test Plan**: 様々な報告日の状態を持つ物件リストレコードを作成し、`calculatePropertyStatus` 関数を呼び出して、返されるステータスが期待と異なることを確認します。未修正コードで実行してバグを観察します。

**Test Cases**:
1. **報告日が今日以前のテスト**: 報告日が今日以前に設定されている物件を作成し、`calculatePropertyStatus` を呼び出す（未修正コードで失敗する）
   - 期待: 「未報告」以外のステータスが返される
   - 実際（未修正）: 「未報告」ステータスが返される（バグ）

2. **報告日が未設定のテスト**: 報告日が null の物件を作成し、`calculatePropertyStatus` を呼び出す（未修正コードで失敗する）
   - 期待: 「未報告」ステータスが返される
   - 実際（未修正）: 「未報告」以外のステータスが返される（バグ）

3. **報告日が未来のテスト**: 報告日が未来の日付の物件を作成し、`calculatePropertyStatus` を呼び出す（未修正コードで失敗する）
   - 期待: 「未報告」ステータスが返される
   - 実際（未修正）: 「未報告」以外のステータスが返される（バグ）

   - 期待: 「未報告」以外のステータスが返される

**Goal**: 修正後のコードで、バグ条件を満たす全ての入力に対して、期待される動作が実現されることを検証します。

**Pseudocode:**
```

  status := calculatePropertyStatus_fixed(listing)
```

### Preservation Checking

**Goal**: 修正後のコードで、バグ条件を満たさない全ての入力に対して、元のコードと同じ結果が返されることを検証します。

**Pseudocode:**
```
FOR ALL listing WHERE NOT isBugCondition(listing) DO
  ASSERT calculatePropertyStatus_original(listing) = calculatePropertyStatus_fixed(listing)
END FOR
```

**Testing Approach**: プロパティベーステストは、保存チェックに推奨されます。理由は以下の通りです：
- 入力ドメイン全体で多数のテストケースを自動生成する
- 手動ユニットテストでは見逃す可能性のあるエッジケースを検出する
- 非バグ入力に対して動作が変更されていないことを強力に保証する

**Test Plan**: まず未修正コードで他のステータス判定ロジックの動作を観察し、次にその動作を捕捉するプロパティベーステストを作成します。
