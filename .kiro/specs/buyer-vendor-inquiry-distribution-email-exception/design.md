# 買主詳細ページ配信メール必須条件バグ修正 Design

## Overview

買主詳細ページの `checkMissingFields` 関数において、「業者問合せ」が「業者問合せ」の場合でも「配信メール」が「要」だと希望条件が必須扱いになるバグを修正します。業者問合せの場合は配信メールを送らないため、希望条件は不要です。

## Glossary

- **Bug_Condition (C)**: 「業者問合せ」が「業者問合せ」であり、かつ「配信メール」が「要」の場合に、希望条件が必須扱いになる条件
- **Property (P)**: 「業者問合せ」が「業者問合せ」の場合、「配信メール」の値に関係なく希望条件を必須項目として扱わない
- **Preservation**: 「業者問合せ」が空欄または「業者問合せ」以外の値の場合の既存の配信メール必須チェックロジックを維持
- **checkMissingFields**: `frontend/frontend/src/pages/BuyerDetailPage.tsx` の関数（340行目付近）で、未入力の必須項目をチェックする
- **broker_inquiry**: 「業者問合せ」フィールド（値: 空欄、「業者問合せ」、「業者（両手）」）
- **distribution_type**: 「配信メール」フィールド（値: 空欄、「要」、「不要」）

## Bug Details

### Bug Condition

バグは、「業者問合せ」が「業者問合せ」であり、かつ「配信メール」が「要」の場合に、希望条件（エリア、希望種別、価格帯）が必須扱いになることで発生します。`checkMissingFields` 関数は `distribution_type === '要'` の場合に無条件で希望条件を必須扱いしており、`broker_inquiry` の値をチェックしていません。

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type { broker_inquiry: string, distribution_type: string }
  OUTPUT: boolean
  
  RETURN input.broker_inquiry === '業者問合せ'
         AND input.distribution_type === '要'
         AND 希望条件が必須扱いされる
END FUNCTION
```

### Examples

- **例1**: 「業者問合せ」が「業者問合せ」、「配信メール」が「要」、希望エリアが未入力 → バリデーションエラーが表示される（期待: エラーなし）
- **例2**: 「業者問合せ」が「業者問合せ」、「配信メール」が「要」、希望種別が未入力 → バリデーションエラーが表示される（期待: エラーなし）
- **例3**: 「業者問合せ」が「業者問合せ」、「配信メール」が「要」、価格帯が未入力 → バリデーションエラーが表示される（期待: エラーなし）
- **エッジケース**: 「業者問合せ」が「業者（両手）」、「配信メール」が「要」、希望条件が未入力 → バリデーションエラーが表示される（期待: エラーあり、既存動作を維持）

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- 「業者問合せ」が空欄または「業者問合せ」以外の値であり、かつ「配信メール」が「要」の場合、希望条件は引き続き必須項目として扱われる
- 「業者問合せ」が空欄または「業者問合せ」以外の値であり、かつ「配信メール」が「不要」の場合、希望条件は引き続き必須項目として扱われない
- 「業者問合せ」が「業者問合せ」の場合、「問合せ元」は引き続き必須項目として扱われない（既存の例外処理を維持）

**Scope:**
「業者問合せ」が「業者問合せ」以外の値（空欄、「業者（両手）」など）の場合は、既存の配信メール必須チェックロジックが完全に維持されます。これには以下が含まれます：
- 「配信メール」が「要」の場合の希望条件必須チェック
- 「配信メール」が「不要」の場合の希望条件任意扱い
- その他の必須項目チェック（初動担当、問合せ元など）

## Hypothesized Root Cause

バグ説明に基づくと、最も可能性の高い原因は：

1. **broker_inquiryチェックの欠如**: `checkMissingFields` 関数の配信メール必須チェックロジック（340行目付近）が `distribution_type === '要'` の場合に無条件で希望条件を必須扱いしており、`broker_inquiry` の値をチェックしていない

2. **既存の例外処理パターンの未適用**: 同じ関数内で「問合せ元」の必須チェックに `broker_inquiry !== '業者問合せ'` の条件が既に実装されているが、配信メール必須チェックには適用されていない

3. **ビジネスロジックの不整合**: 業者問合せの場合は配信メールを送らないというビジネスルールが、コードに反映されていない

## Correctness Properties

Property 1: Bug Condition - 業者問合せ時の配信メール必須除外

_For any_ 買主データにおいて「業者問合せ」が「業者問合せ」の場合、修正後の checkMissingFields 関数 SHALL 「配信メール」の値に関係なく希望条件（エリア、希望種別、価格帯）を必須項目として扱わず、バリデーションエラーを表示しない。

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - 非業者問合せ時の既存動作維持

_For any_ 買主データにおいて「業者問合せ」が「業者問合せ」以外の値（空欄、「業者（両手）」など）の場合、修正後の checkMissingFields 関数 SHALL 既存の配信メール必須チェックロジックを完全に維持し、「配信メール」が「要」の場合は希望条件を必須項目として扱い、「不要」の場合は任意扱いする。

**Validates: Requirements 3.1, 3.2, 3.3**

## Fix Implementation

### Changes Required

根本原因分析が正しいと仮定すると：

**File**: `frontend/frontend/src/pages/BuyerDetailPage.tsx`

**Function**: `checkMissingFields`（340行目付近）

**Specific Changes**:
1. **配信メール必須チェックの条件追加**: 既存の `distribution_type === '要'` の条件に `broker_inquiry !== '業者問合せ'` を追加
   - 実装: `if (buyer.broker_inquiry !== '業者問合せ' && buyer.distribution_type && String(buyer.distribution_type).trim() === '要') { ... }`
   - 既存の「問合せ元」例外処理と同じパターンを適用

2. **コメントの追加**: 変更箇所に業者問合せ時の例外処理であることを明記
   - 実装: `// 業者問合せの場合は配信メールを送らないため、希望条件は不要`

3. **既存ロジックの維持**: 希望条件の必須チェックロジック（エリア、希望種別、価格帯）は変更せず、条件分岐のみ追加

## Testing Strategy

### Validation Approach

テスト戦略は2段階アプローチに従います：まず、未修正コードでバグを再現する探索的テストを実行し、次に修正後のコードで正しい動作と既存動作の保持を検証します。

### Exploratory Bug Condition Checking

**Goal**: 修正前のコードでバグを再現し、根本原因分析を確認または反証します。反証された場合は、再仮説が必要です。

**Test Plan**: 「業者問合せ」が「業者問合せ」であり、かつ「配信メール」が「要」の買主データで `checkMissingFields` 関数を呼び出し、希望条件が必須扱いされることを確認します。未修正コードで実行してバグを観察し、根本原因を理解します。

**Test Cases**:
1. **業者問合せ + 配信メール要 + 希望エリア未入力**: 「業者問合せ」が「業者問合せ」、「配信メール」が「要」、希望エリアが未入力の場合、バリデーションエラーが表示される（未修正コードで失敗）
2. **業者問合せ + 配信メール要 + 希望種別未入力**: 「業者問合せ」が「業者問合せ」、「配信メール」が「要」、希望種別が未入力の場合、バリデーションエラーが表示される（未修正コードで失敗）
3. **業者問合せ + 配信メール要 + 価格帯未入力**: 「業者問合せ」が「業者問合せ」、「配信メール」が「要」、価格帯が未入力の場合、バリデーションエラーが表示される（未修正コードで失敗）
4. **業者問合せ + 配信メール不要 + 希望条件未入力**: 「業者問合せ」が「業者問合せ」、「配信メール」が「不要」、希望条件が未入力の場合、バリデーションエラーが表示されない（未修正コードでも成功、エッジケース確認）

**Expected Counterexamples**:
- 「業者問合せ」が「業者問合せ」であり、かつ「配信メール」が「要」の場合、希望条件が必須扱いされる
- 可能な原因: `broker_inquiry` チェックの欠如、既存の例外処理パターンの未適用

### Fix Checking

**Goal**: バグ条件を満たす全ての入力に対して、修正後の関数が期待される動作を生成することを検証します。

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := checkMissingFields_fixed(input)
  ASSERT NOT result.includes('desired_area')
  ASSERT NOT result.includes('desired_property_type')
  ASSERT NOT result.includes('price_range_*')
END FOR
```

### Preservation Checking

**Goal**: バグ条件を満たさない全ての入力に対して、修正後の関数が元の関数と同じ結果を生成することを検証します。

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT checkMissingFields_original(input) = checkMissingFields_fixed(input)
END FOR
```

**Testing Approach**: プロパティベーステストは保存チェックに推奨されます。理由：
- 入力ドメイン全体で多くのテストケースを自動生成
- 手動ユニットテストが見逃す可能性のあるエッジケースをキャッチ
- 非バグ入力に対して動作が変更されていないことを強力に保証

**Test Plan**: まず未修正コードで非業者問合せ入力の動作を観察し、次にその動作をキャプチャするプロパティベーステストを作成します。

**Test Cases**:
