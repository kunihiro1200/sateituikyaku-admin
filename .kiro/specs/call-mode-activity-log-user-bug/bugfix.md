# バグ修正要件ドキュメント

## はじめに

売主リスト・買主リスト・物件リストの通話モードページおよびその他のアクション（メール送信・SMS送信・電話発信）を実行すると、アクティビティログ・通話履歴に記録されるユーザー名（イニシャル）が全員間違っている。例えば「yurine〜」のアカウントで操作しているのに「U（裏〜）」として記録される。全ユーザー・全リスト・全アクションで同様の問題が発生しており、Supabaseのemployeesテーブルのデータ自体は正しいため、システム共通のログインユーザー取得ロジックに根本的な問題がある。

## バグ分析

### 現在の動作（不具合）

1.1 WHEN ログインユーザーAが通話モードページでメール送信アクションを実行する THEN システムはアクティビティログにユーザーAとは異なる別のユーザーのイニシャルを記録する

1.2 WHEN ログインユーザーAがSMS送信アクションを実行する THEN システムはアクティビティログにユーザーAとは異なる別のユーザーのイニシャルを記録する

1.3 WHEN ログインユーザーAが通話モードページのヘッダーにある電話番号をクリックしてアプリで電話をかける THEN システムは通話履歴にユーザーAとは異なる別のユーザーのイニシャルを記録する

1.4 WHEN 複数の異なるユーザーがそれぞれ通話モードページでアクションを実行する THEN システムは全員のアクティビティログ・通話履歴に間違ったイニシャルを記録する（特定のユーザーだけでなく全員が対象）

1.5 WHEN 売主リスト・買主リスト・物件リストのいずれかの通話モードページでアクションを実行する THEN システムはいずれのリストにおいても同じ間違ったユーザーのイニシャルを記録する（特定のリストだけでなく全リストが対象）

1.6 WHEN バックエンドの`/api/employees/initials-by-email`エンドポイントが呼ばれる THEN システムはセッションキャッシュ（5分TTL）に保存された古いemployeeオブジェクトを使用してイニシャルを解決するため、キャッシュが古い場合に誤ったイニシャルが返される可能性がある

1.7 WHEN バックエンドの`send-template-email`エンドポイントが`senderInitials`を解決する THEN システムは`req.employee.initials`を参照するが、`validateSession`のキャッシュに`initials`カラムが含まれていない場合、または古いキャッシュが使われている場合に誤ったイニシャルが返される

### 期待される動作（正しい動作）

2.1 WHEN ログインユーザーAが通話モードページでメール送信アクションを実行する THEN システムはアクティビティログにユーザーAの正しいイニシャルを記録する

2.2 WHEN ログインユーザーAがSMS送信アクションを実行する THEN システムはアクティビティログにユーザーAの正しいイニシャルを記録する

2.3 WHEN ログインユーザーAが通話モードページのヘッダーにある電話番号をクリックしてアプリで電話をかける THEN システムは通話履歴にユーザーAの正しいイニシャルを記録する

2.4 WHEN 複数の異なるユーザーがそれぞれ通話モードページでアクションを実行する THEN システムは各ユーザーのアクティビティログ・通話履歴にそれぞれの正しいイニシャルを記録する

2.5 WHEN 売主リスト・買主リスト・物件リストのいずれかの通話モードページでアクションを実行する THEN システムはいずれのリストにおいても実行したログインユーザーの正しいイニシャルを記録する

2.6 WHEN バックエンドの`/api/employees/initials-by-email`エンドポイントが呼ばれる THEN システムはリクエストのJWTトークンから取得したメールアドレスを使ってemployeesテーブルを検索し、現在ログインしているユーザーの正しいイニシャルを返す

2.7 WHEN バックエンドの`send-template-email`エンドポイントが`senderInitials`を解決する THEN システムはリクエストのJWTトークンから特定されたログインユーザーのメールアドレスを使ってemployeesテーブルから正しいイニシャルを取得する

### 変更してはいけない動作（リグレッション防止）

3.1 WHEN ログインユーザーが通話モードページでメール送信アクションを実行する THEN システムはSHALL CONTINUE TO メール送信自体は正常に完了させる（ログ記録の修正がメール送信機能に影響しない）

3.2 WHEN ログインユーザーがSMS送信アクションを実行する THEN システムはSHALL CONTINUE TO SMSアプリを正常に開く（ログ記録の修正がSMS機能に影響しない）

3.3 WHEN ログインユーザーが通話モードページのヘッダーから電話をかける THEN システムはSHALL CONTINUE TO 電話アプリを正常に起動させる（通話履歴記録の修正が電話機能に影響しない）

3.4 WHEN ログインユーザーが通話モードページでアクションを実行する THEN システムはSHALL CONTINUE TO アクティビティログの内容（メール件名・テンプレート名など）は正しく記録する

3.5 WHEN employeesテーブルにinitialsカラムが設定されているユーザーがログインする THEN システムはSHALL CONTINUE TO そのユーザーの認証・セッション管理は正常に動作する

3.6 WHEN `/api/employees/initials-by-email`エンドポイントが呼ばれる THEN システムはSHALL CONTINUE TO DBにinitialsが存在する場合はDBの値を優先し、存在しない場合はスプレッドシートにフォールバックする既存の優先順位ロジックを維持する

3.7 WHEN 売主リスト・買主リスト・物件リストのいずれかでアクションを実行する THEN システムはSHALL CONTINUE TO 各リストの既存機能（一覧表示・詳細表示・検索など）は正常に動作する
