# 買主候補パフォーマンス修正 設計ドキュメント

## 概要

物件詳細画面（売主管理システム）のヘッダーにある「買主候補」ボタンをクリックすると、約50秒かかって買主候補一覧が表示される問題を修正します。また、新しいタブで開かれるウィンドウのサイズが小さく使いにくい問題も修正します。

修正アプローチ：
1. **パフォーマンス問題**: `BuyerCandidateService.getCandidatesForProperty()`メソッドの処理を最適化（N+1クエリ問題の解消、不要な処理の削除）
2. **表示サイズ問題**: `window.open()`の第3引数にウィンドウサイズを指定

## 用語集

- **Bug_Condition (C)**: 買主候補ボタンをクリックした時に、約50秒の遅延が発生する条件、または新しいタブのウィンドウサイズが小さい条件
- **Property (P)**: 買主候補ボタンをクリックした時に、3秒以内に買主候補一覧が表示され、適切なサイズのウィンドウで表示される
- **Preservation**: マウスクリックによるボタン操作、買主候補の抽出条件、テーブル表示形式、新しいタブで開く動作が変わらないこと
- **getCandidatesForProperty**: `backend/src/services/BuyerCandidateService.ts`のメソッドで、物件番号に対する買主候補を抽出する
- **getPropertyAddressesInBatch**: 複数の物件番号に対して住所と価格を一括取得するメソッド
- **N+1クエリ問題**: 各買主の問い合わせ物件住所を取得する際に、個別にクエリを発行していた問題（現在は一括取得に修正済み）

## バグ詳細

### バグ条件

#### パフォーマンス問題

買主候補ボタンをクリックした時に、`BuyerCandidateService.getCandidatesForProperty()`メソッドが約50秒かかる。原因は以下の可能性がある：

1. **買主の全件取得**: `buyers`テーブルから全件取得（ページネーション処理あり）
2. **フィルタリング処理**: 各買主に対してエリアマッチング、物件種別マッチング、価格マッチングを実行
3. **距離マッチング**: 物件座標と買主の希望エリアの距離を計算（非同期処理）
4. **問い合わせ物件住所の取得**: 各買主の問い合わせ物件番号から住所を取得（一括取得に修正済み）

**形式的仕様**:
```
FUNCTION isBugCondition_Performance(input)
  INPUT: input of type ButtonClickEvent
  OUTPUT: boolean
  
  RETURN input.buttonId == 'buyer-candidates-button'
         AND input.propertyNumber EXISTS
         AND responseTime > 3000ms  // 3秒以上
END FUNCTION
```

#### 表示サイズ問題

買主候補ボタンをクリックして新しいタブが開かれた時に、ウィンドウサイズが小さい（デフォルトサイズ）。`window.open()`の第3引数が指定されていないため、ブラウザのデフォルトサイズで開かれる。

**形式的仕様**:
```
FUNCTION isBugCondition_WindowSize(input)
  INPUT: input of type ButtonClickEvent
  OUTPUT: boolean
  
  RETURN input.buttonId == 'buyer-candidates-button'
         AND input.propertyNumber EXISTS
         AND windowSize < EXPECTED_SIZE  // 適切なサイズより小さい
END FUNCTION
```

### 例

#### パフォーマンス問題の例
- **例1**: 物件AA9926の買主候補ボタンをクリック → 約50秒後に10件の買主候補が表示される（期待: 3秒以内）
- **例2**: 物件BB1234の買主候補ボタンをクリック → 約50秒後に5件の買主候補が表示される（期待: 3秒以内）
- **例3**: 物件CC5678の買主候補ボタンをクリック → 約50秒後に20件の買主候補が表示される（期待: 3秒以内）

#### 表示サイズ問題の例
- **例1**: 買主候補ボタンをクリック → 小さいウィンドウで買主候補一覧が表示される（期待: 適切なサイズのウィンドウ）
- **例2**: 買主候補ボタンをクリック → テーブルが見切れて表示される（期待: テーブル全体が見えるサイズ）

## 期待される動作

### 保存要件

**変更されない動作**:
- マウスクリックによる買主候補ボタンの操作は引き続き正常に動作する
- 買主候補の抽出条件（エリアマッチング、物件種別マッチング、価格マッチング）は変わらない
- 買主候補一覧のテーブル表示形式は変わらない
- 買主候補一覧は引き続き新しいタブで開かれる
- 物件詳細画面の他のボタンやフィールドの操作は引き続き正常に動作する
- 買主候補一覧以外のページは引き続き正常に動作する

**スコープ**:
買主候補ボタン以外の操作（物件詳細画面の他のボタン、フィールド編集、他のページ）は完全に影響を受けない。

## 仮説的根本原因

コードベースの分析に基づく、最も可能性の高い原因：

### 1. **買主の全件取得とフィルタリング処理**
`getCandidatesForProperty()`メソッドが`buyers`テーブルから全件取得（ページネーション処理あり）し、その後フィルタリング処理を実行している。買主が数千件ある場合、この処理に時間がかかる可能性がある。

**詳細**:
- `buyers`テーブルから`distribution_type = '要'`の買主を全件取得
- 各買主に対してエリアマッチング、物件種別マッチング、価格マッチングを実行
- フィルタリング処理がJavaScriptで実行されるため、買主が多い場合に遅延が発生

### 2. **距離マッチング処理**
`filterCandidatesAsync()`メソッドが各買主に対して距離マッチングを実行している。この処理が非同期で実行されるため、買主が多い場合に遅延が発生する可能性がある。

**詳細**:
- `matchesAreaCriteriaWithDistance()`メソッドが各買主に対して呼び出される
- 物件座標と買主の希望エリアの距離を計算
- 非同期処理のため、買主が多い場合に遅延が発生

### 3. **問い合わせ物件住所の取得**
各買主の問い合わせ物件番号から住所を取得する処理が、一括取得に修正済みだが、物件番号が多い場合に遅延が発生する可能性がある。

**詳細**:
- `getPropertyAddressesInBatch()`メソッドが複数の物件番号に対して住所と価格を一括取得
- 物件番号が多い場合、Supabaseクエリに時間がかかる可能性がある

### 4. **ウィンドウサイズの指定なし**
`window.open()`の第3引数が指定されていないため、ブラウザのデフォルトサイズで開かれる。

**詳細**:
- `PropertyListingDetailPage.tsx`の`handleOpenBuyerCandidates()`メソッドで`window.open()`を呼び出し
- 第3引数が`'noopener,noreferrer'`のみで、ウィンドウサイズが指定されていない

## 正確性プロパティ

Property 1: Bug Condition - 買主候補ボタンのパフォーマンス改善

_For any_ 買主候補ボタンのクリックイベントにおいて、物件番号が存在する場合、修正後の`getCandidatesForProperty()`メソッドは3秒以内に買主候補一覧を返し、ブラウザは適切なサイズのウィンドウで買主候補一覧を表示する。

**Validates: Requirements 2.1, 2.2, 2.3**

Property 2: Preservation - 既存機能の保持

_For any_ 買主候補ボタン以外の操作（物件詳細画面の他のボタン、フィールド編集、他のページ）において、修正後のコードは修正前のコードと同じ動作を生成し、既存機能を保持する。

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5**

## 修正実装

### 必要な変更

根本原因分析が正しいと仮定した場合：

**ファイル**: `backend/src/services/BuyerCandidateService.ts`

**メソッド**: `getCandidatesForProperty()`

**具体的な変更**:
1. **DBレベルでのフィルタリング強化**: Supabaseクエリに`.eq('distribution_type', '要')`、`.not('latest_status', 'like', '%買付%')`、`.not('latest_status', 'like', '%D%')`を追加（既に実装済み）
2. **インデックスの追加**: `buyers`テーブルの`distribution_type`、`latest_status`、`deleted_at`カラムにインデックスを追加（要確認）
3. **フィルタリング処理の最適化**: JavaScriptでのフィルタリング処理を最小限にする
4. **距離マッチング処理の最適化**: 距離マッチングが本当に必要か確認し、不要であれば削除
5. **問い合わせ物件住所の取得最適化**: `getPropertyAddressesInBatch()`メソッドのクエリを最適化

**ファイル**: `frontend/frontend/src/pages/PropertyListingDetailPage.tsx`

**メソッド**: `handleOpenBuyerCandidates()`

**具体的な変更**:
1. **ウィンドウサイズの指定**: `window.open()`の第3引数に`'width=1400,height=900,noopener,noreferrer'`を指定

## テスト戦略

### 検証アプローチ

テスト戦略は2段階アプローチに従います：まず、修正前のコードでバグを再現する探索的バグ条件チェックを実行し、次に修正が正しく動作し、既存の動作を保持することを検証します。

### 探索的バグ条件チェック

**目標**: 修正を実装する前に、修正前のコードでバグを再現する。根本原因分析を確認または反証する。反証された場合、再仮説を立てる必要がある。

**テスト計画**: 買主候補ボタンをクリックし、レスポンスタイムを測定するテストを作成する。修正前のコードで実行し、約50秒の遅延を観察する。

**テストケース**:
1. **パフォーマンステスト**: 物件AA9926の買主候補ボタンをクリック → レスポンスタイムを測定（修正前のコードで失敗）
2. **買主数テスト**: 買主が10件、50件、100件の場合のレスポンスタイムを測定（修正前のコードで失敗）
3. **距離マッチングテスト**: 距離マッチング処理の実行時間を測定（修正前のコードで失敗の可能性）
4. **ウィンドウサイズテスト**: 買主候補ボタンをクリック → ウィンドウサイズを測定（修正前のコードで失敗）

**期待される反例**:
- レスポンスタイムが3秒を超える（約50秒）
- 可能性のある原因: 買主の全件取得とフィルタリング処理、距離マッチング処理、問い合わせ物件住所の取得

### 修正チェック

**目標**: バグ条件が成立するすべての入力に対して、修正後の関数が期待される動作を生成することを検証する。

**疑似コード**:
```
FOR ALL input WHERE isBugCondition_Performance(input) DO
  result := getCandidatesForProperty_fixed(input.propertyNumber)
  ASSERT responseTime(result) <= 3000ms
END FOR

FOR ALL input WHERE isBugCondition_WindowSize(input) DO
  result := handleOpenBuyerCandidates_fixed(input)
  ASSERT windowSize(result) >= EXPECTED_SIZE
END FOR
```

### 保存チェック

**目標**: バグ条件が成立しないすべての入力に対して、修正後の関数が修正前の関数と同じ結果を生成することを検証する。

**疑似コード**:
```
FOR ALL input WHERE NOT isBugCondition_Performance(input) AND NOT isBugCondition_WindowSize(input) DO
  ASSERT originalFunction(input) = fixedFunction(input)
END FOR
```

**テストアプローチ**: プロパティベーステストは保存チェックに推奨されます。理由：
- 入力ドメイン全体で多くのテストケースを自動的に生成する
- 手動ユニットテストが見逃す可能性のあるエッジケースをキャッチする
- 非バグ入力に対して動作が変わらないことを強力に保証する

**テスト計画**: 修正前のコードで買主候補ボタン以外の操作の動作を観察し、その動作をキャプチャするプロパティベーステストを作成する。

**テストケース**:
1. **買主候補抽出条件の保存**: 修正前と修正後で同じ買主候補が抽出されることを確認
2. **テーブル表示形式の保存**: 修正前と修正後で同じテーブル表示形式が使用されることを確認
3. **新しいタブで開く動作の保存**: 修正前と修正後で新しいタブで開かれることを確認
4. **他のボタンの動作の保存**: 物件詳細画面の他のボタンが引き続き正常に動作することを確認
