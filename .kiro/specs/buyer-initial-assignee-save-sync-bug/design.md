# 買主初動担当保存時スプレッドシート同期エラー バグ修正設計

## Overview

買主詳細画面で初動担当フィールドに「久」を選択して保存ボタンを押すと、DBへの保存は成功するがスプレッドシートへの同期に失敗する問題を修正します。エラーログには404エラー（`/api/buyers/7260/related`）、409エラー（`/api/buyers/7260?sync=true`）、および非同期メッセージチャネルエラーが記録されています。

この問題により、DBとスプレッドシートのデータ不整合が発生し、ユーザーがスプレッドシートで最新の初動担当情報を確認できない状態になっています。

## Glossary

- **Bug_Condition (C)**: 買主番号7260の初動担当フィールドに「久」を選択して保存ボタンを押す操作
- **Property (P)**: DBへの保存とスプレッドシートへの同期の両方が成功し、エラーメッセージが表示されない
- **Preservation**: 他の買主番号、他のフィールド、他の初動担当値での保存が引き続き正常に動作すること
- **updateWithSync**: `BuyerService.ts`の`updateWithSync`メソッド。買主データをDBに保存し、スプレッドシートに同期する
- **buyerApi.update**: フロントエンドの`api.ts`の`buyerApi.update`メソッド。`sync: true`オプションで`PUT /api/buyers/:id?sync=true`を呼び出す
- **RelatedBuyerService**: 関連買主を検索するサービス。`/api/buyers/:id/related`エンドポイントで使用される

## Bug Details

### Bug Condition


バグは、買主番号7260の初動担当フィールドに「久」を選択して保存ボタンを押したときに発生します。`BuyerDetailPage.tsx`の`handleSectionSave`関数が`buyerApi.update(buyer_number, changedFields, { sync: true })`を呼び出し、バックエンドの`PUT /api/buyers/:id?sync=true`エンドポイントが`BuyerService.updateWithSync()`を実行します。

DBへの保存は成功しますが、その後のスプレッドシート同期処理で以下のエラーが発生します：
- 404エラー: `/api/buyers/7260/related`（関連買主取得の失敗）
- 409エラー: `/api/buyers/7260?sync=true`（競合検出）
- 非同期メッセージチャネルエラー（ブラウザ拡張機能またはフロントエンド処理の問題）

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type { buyerNumber: string, fieldName: string, newValue: string }
  OUTPUT: boolean
  
  RETURN input.buyerNumber == '7260'
         AND input.fieldName == 'initial_assignee'
         AND input.newValue == '久'
         AND saveButtonPressed()
END FUNCTION
```

### Examples

- **例1**: 買主番号7260の初動担当フィールドに「久」を選択して保存 → DBへの保存は成功するが、スプレッドシート同期に失敗し、警告メッセージ「DBへの保存は完了しましたが、スプレッドシートへの同期に失敗しました」が表示される
- **例2**: 買主番号7260の初動担当フィールドに「Y」を選択して保存 → DBへの保存とスプレッドシート同期の両方が成功する（バグ条件を満たさない）
- **例3**: 買主番号4370の初動担当フィールドに「久」を選択して保存 → DBへの保存とスプレッドシート同期の両方が成功する（バグ条件を満たさない）
- **エッジケース**: 買主番号7260の問合せ元フィールドを編集して保存 → DBへの保存とスプレッドシート同期の両方が成功する（初動担当以外のフィールド）

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- 買主番号7260以外の買主で初動担当フィールドを編集して保存した場合、DBへの保存とスプレッドシート同期の両方が引き続き正常に実行される
- 買主番号7260で初動担当以外のフィールド（問合せ元、最新状況など）を編集して保存した場合、DBへの保存とスプレッドシート同期の両方が引き続き正常に実行される
- 買主番号7260の初動担当フィールドに「久」以外の値（「Y」「I」など）を選択して保存した場合、DBへの保存とスプレッドシート同期の両方が引き続き正常に実行される

**Scope:**
買主番号7260の初動担当フィールドに「久」を選択して保存する操作以外の全ての保存操作は、完全に影響を受けないようにする必要があります。これには以下が含まれます：
- 他の買主番号での初動担当フィールドの保存
- 買主番号7260での他のフィールドの保存
- 買主番号7260の初動担当フィールドに他の値を保存

## Hypothesized Root Cause

バグ記述とエラーログに基づき、最も可能性の高い原因は以下の通りです：

1. **買主番号7260のデータ不整合**: 買主番号7260のレコードに`buyer_id`（UUID）が存在しないか、不正な値が設定されている可能性
   - `/api/buyers/7260/related`エンドポイントは、まず`getByBuyerNumber('7260')`で買主を取得し、その`buyer_id`を使って関連買主を検索する
   - `buyer_id`が存在しないか不正な場合、404エラーが発生する

2. **スプレッドシート同期時の競合検出ロジックの誤動作**: `updateWithSync`メソッドの競合チェックが、買主番号7260の初動担当フィールドに対して誤って競合を検出している可能性
   - `last_synced_at`が存在する場合、スプレッドシートの現在値とDBの期待値を比較する
   - 買主番号7260の初動担当フィールドのスプレッドシート値が、DBの期待値と異なる場合、409エラーが発生する

3. **スプレッドシート書き込みサービスのエラー**: `BuyerSpreadsheetWriteService.updateFields()`が買主番号7260の初動担当フィールドの更新に失敗している可能性
   - スプレッドシートの行が見つからない
   - 初動担当カラムのマッピングが正しくない
   - Google Sheets APIのエラー

4. **非同期メッセージチャネルエラー**: フロントエンドのブラウザ拡張機能またはサービスワーカーが、APIレスポンスの受信前にメッセージチャネルを閉じている可能性
   - これはブラウザ側の問題であり、バックエンドの修正では解決できない可能性がある

## Correctness Properties

Property 1: Bug Condition - 初動担当フィールド保存時の同期成功

_For any_ 買主番号7260の初動担当フィールドに「久」を選択して保存ボタンを押す操作において、修正後のシステムはDBへの保存とスプレッドシートへの同期の両方を成功させ、エラーメッセージを表示しない。

**Validates: Requirements 2.1, 2.2, 2.3, 2.4**

Property 2: Preservation - 他の保存操作の動作維持

_For any_ 買主番号7260の初動担当フィールドに「久」を選択して保存する操作以外の保存操作（他の買主番号、他のフィールド、他の初動担当値）において、修正後のシステムは修正前と同じ動作を維持し、DBへの保存とスプレッドシート同期の両方を引き続き正常に実行する。

**Validates: Requirements 3.1, 3.2, 3.3, 3.4**

## Fix Implementation

### Changes Required

根本原因分析が正しいと仮定した場合の修正内容：

**File**: `backend/src/services/BuyerService.ts`

**Method**: `getByBuyerNumber()`

**Specific Changes**:
1. **データ整合性チェックの追加**: `getByBuyerNumber()`メソッドで取得した買主レコードに`buyer_id`が存在するか確認し、存在しない場合はエラーログを出力
   - 買主番号7260のレコードに`buyer_id`が存在しない場合、データ修復が必要
   - `buyer_id`はUUID型の主キーであり、全ての買主レコードに必須

2. **競合検出ロジックの修正**: `updateWithSync()`メソッドの競合チェックで、買主番号7260の初動担当フィールドに対して誤った競合検出が発生している場合、ロジックを修正
   - `last_synced_at`が存在しない場合は競合チェックをスキップ（既に実装済み）
   - スプレッドシートの現在値取得時のエラーハンドリングを強化

3. **スプレッドシート書き込みエラーハンドリングの強化**: `BuyerSpreadsheetWriteService.updateFields()`で買主番号7260の行が見つからない場合のエラーハンドリングを追加
   - 行が見つからない場合は、詳細なエラーログを出力
   - 初動担当カラムのマッピングが正しいか確認

4. **404エラーの根本原因調査**: `/api/buyers/7260/related`エンドポイントで404エラーが発生する原因を特定
   - `getByBuyerNumber('7260')`が`null`を返している可能性
   - 買主番号7260のレコードが`deleted_at`でソフトデリートされている可能性
   - データベースに買主番号7260のレコードが存在しない可能性

5. **データ修復スクリプトの作成**: 買主番号7260のデータ不整合を修復するスクリプトを作成
   - `buyer_id`が存在しない場合は、UUIDを生成して設定
   - `deleted_at`が設定されている場合は、`null`に戻す

**File**: `backend/src/services/BuyerSpreadsheetWriteService.ts`

**Method**: `updateFields()`

**Specific Changes**:
1. **エラーログの強化**: 買主番号7260の行が見つからない場合、詳細なエラーログを出力
2. **初動担当カラムマッピングの確認**: `buyer-column-mapping.json`で初動担当カラムが正しくマッピングされているか確認

## Testing Strategy

### Validation Approach

テスト戦略は2段階アプローチに従います：まず、未修正コードでバグを再現する探索的テストを実行し、次に修正後のコードで修正が正しく機能し、既存の動作が保持されていることを検証します。

### Exploratory Bug Condition Checking

**Goal**: 修正を実装する前に、未修正コードでバグを再現する。根本原因分析を確認または反証する。反証した場合は、再仮説が必要。

**Test Plan**: 買主番号7260の初動担当フィールドに「久」を選択して保存する操作をシミュレートするテストを作成します。未修正コードで実行してエラーを観察し、根本原因を理解します。

**Test Cases**:
1. **買主番号7260の存在確認**: 買主番号7260がDBに存在するか確認（未修正コードで失敗する可能性）
2. **buyer_id存在確認**: 買主番号7260のレコードに`buyer_id`が存在するか確認（未修正コードで失敗する可能性）
3. **関連買主取得テスト**: `/api/buyers/7260/related`エンドポイントを呼び出して404エラーを再現（未修正コードで失敗）
4. **初動担当保存テスト**: 買主番号7260の初動担当フィールドに「久」を保存して409エラーを再現（未修正コードで失敗）

**Expected Counterexamples**:
- 買主番号7260のレコードに`buyer_id`が存在しない、または不正な値が設定されている
- `/api/buyers/7260/related`エンドポイントが404エラーを返す
- `updateWithSync()`メソッドが409エラーを返す
- 可能性のある原因: データ不整合、競合検出ロジックの誤動作、スプレッドシート書き込みエラー

### Fix Checking

**Goal**: バグ条件を満たす全ての入力に対して、修正後の関数が期待される動作を生成することを検証する。

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := updateWithSync_fixed(input)
  ASSERT expectedBehavior(result)
END FOR
```

### Preservation Checking

**Goal**: バグ条件を満たさない全ての入力に対して、修正後の関数が元の関数と同じ結果を生成することを検証する。

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT updateWithSync_original(input) = updateWithSync_fixed(input)
END FOR
```

**Testing Approach**: プロパティベーステストは、保存チェックに推奨されます。理由：
- 入力ドメイン全体で多数のテストケースを自動生成する
- 手動ユニットテストが見逃す可能性のあるエッジケースを捕捉する
- 非バグ入力に対して動作が変更されていないことを強力に保証する

**Test Plan**: まず未修正コードで他の買主番号・他のフィールドの保存動作を観察し、その動作を捕捉するプロパティベーステストを作成します。

**Test Cases**:
1. **他の買主番号での保存**: 未修正コードで買主番号4370の初動担当フィールドに「久」を保存して正常に動作することを観察し、修正後も同じ動作を維持することをテスト
2. **他のフィールドの保存**: 未修正コードで買主番号7260の問合せ元フィールドを保存して正常に動作することを観察し、修正後も同じ動作を維持することをテスト
3. **他の初動担当値の保存**: 未修正コードで買主番号7260の初動担当フィールドに「Y」を保存して正常に動作することを観察し、修正後も同じ動作を維持することをテスト
4. **保存ボタンの有効化**: 未修正コードで初動担当フィールドを変更したときに保存ボタンが有効化されることを観察し、修正後も同じ動作を維持することをテスト
