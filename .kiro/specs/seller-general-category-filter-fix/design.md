# 売主リスト「一般」カテゴリフィルタ修正 設計ドキュメント

## Overview

売主リストのサイドバーで「一般（23）」カテゴリをクリックすると、カウント表示は23件だが、実際には全件が展開リストに表示されてしまうバグを修正します。

根本原因は、バックエンドの `SellerService.supabase.ts` の `case 'general':` のフィルタリング条件が間違っていることです。現在の条件は「次電日が今日より後」（`.gt('next_call_date', todayJST)`）となっていますが、正しくは「次電日が今日ではない」（`.neq('next_call_date', todayJST)`）である必要があります。

GAS側のカウント計算ロジックは正しく「次電日 ≠ 今日」となっているため、バックエンドのフィルタリング条件のみを修正します。

## Glossary

- **Bug_Condition (C)**: バックエンドのフィルタリング条件が間違っている状態 - 「次電日が今日より後」という条件を使用している
- **Property (P)**: 正しいフィルタリング動作 - 「次電日が今日ではない」という条件を使用し、GASのカウント計算と一致する結果を返す
- **Preservation**: 他のサイドバーカテゴリ（専任、訪問後他決など）のフィルタリング動作が変更されないこと
- **SellerService.supabase.ts**: `backend/src/services/SellerService.supabase.ts` - 売主管理システム用のバックエンドサービス
- **listSellers()**: 売主リストを取得するメソッド - サイドバーカテゴリに基づいてフィルタリングを実行
- **todayJST**: 日本標準時（JST）の今日の日付（YYYY-MM-DD形式）

## Bug Details

### Bug Condition

バグは、ユーザーがサイドバーの「一般（23）」カテゴリをクリックしたときに発生します。`listSellers()` メソッドは、`case 'general':` ブロックで `.gt('next_call_date', todayJST)` という間違った条件を使用しているため、「次電日が今日より後」の売主のみを返します。

しかし、GAS側のカウント計算ロジックは「次電日 ≠ 今日」（`nextCallDate !== todayStr`）という正しい条件を使用しているため、カウント（23件）とフィルタリング結果が一致しません。

**Formal Specification:**
```
FUNCTION isBugCondition(input)
  INPUT: input of type { statusCategory: string, todayJST: string }
  OUTPUT: boolean
  
  RETURN input.statusCategory === 'general'
         AND filterCondition === '.gt("next_call_date", todayJST)'
         AND GASCountCondition === 'nextCallDate !== todayStr'
END FUNCTION
```

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- 他のサイドバーカテゴリ（専任、訪問後他決など）のフィルタリング動作は変更されない
- GAS側のカウント計算ロジックは変更されない
- 一般カテゴリ以外のフィルタリング条件は変更されない

**Scope:**
「一般」カテゴリ以外のサイドバーカテゴリをクリックした場合、フィルタリング動作は完全に影響を受けません。これには以下が含まれます：
- 専任カテゴリ
- 訪問後他決カテゴリ
- 当日TEL分カテゴリ
- その他全てのサイドバーカテゴリ

## Hypothesized Root Cause

バグ説明に基づくと、最も可能性の高い問題は以下の通りです：

1. **間違ったフィルタリング条件**: `case 'general':` ブロックで `.gt('next_call_date', todayJST)` を使用している
   - 正しい条件は `.neq('next_call_date', todayJST)` である必要がある
   - GAS側のカウント計算ロジックは `nextCallDate !== todayStr` を使用している

2. **GASとバックエンドの条件不一致**: GAS側は「次電日 ≠ 今日」を使用しているが、バックエンド側は「次電日 > 今日」を使用している
   - この不一致により、カウントとフィルタリング結果が一致しない

3. **コピー＆ペーストエラー**: 他のカテゴリ（専任、訪問後他決）も `.gt('next_call_date', todayJST)` を使用している
   - 一般カテゴリも同じ条件をコピーしてしまった可能性がある

## Correctness Properties

Property 1: Bug Condition - 一般カテゴリのフィルタリング条件修正

_For any_ 売主リストのフィルタリングリクエストで、サイドバーカテゴリが「一般」であり、次電日が今日ではない（今日より前または今日より後）場合、修正後の `listSellers()` メソッドは、その売主をフィルタリング結果に含め、GASのカウント計算と一致する結果を返すこと。

**Validates: Requirements 2.1, 2.2**

Property 2: Preservation - 他のカテゴリのフィルタリング動作

_For any_ 売主リストのフィルタリングリクエストで、サイドバーカテゴリが「一般」ではない場合、修正後のコードは、修正前のコードと全く同じフィルタリング結果を返し、他のカテゴリのフィルタリング動作を保持すること。

**Validates: Requirements 3.1, 3.2, 3.3**

## Fix Implementation

### Changes Required

根本原因分析が正しいと仮定すると：

**File**: `backend/src/services/SellerService.supabase.ts`

**Function**: `listSellers()`

**Specific Changes**:
1. **フィルタリング条件の修正**: `case 'general':` ブロックの条件を変更
   - 変更前: `.gt('next_call_date', todayJST)`
   - 変更後: `.neq('next_call_date', todayJST)`

2. **コメントの更新**: 条件の説明を正確にする
   - 変更前: `// 一般カテゴリー（専任他決打合せ <> "完了" + 次電日 <> TODAY() + 状況が一般媒介 + 契約年月 >= 2025/6/23）`
   - 変更後: `// 一般カテゴリー（専任他決打合せ <> "完了" + 次電日 <> TODAY() + 状況が一般媒介 + 契約年月 >= 2025/6/23）`（変更なし、既に正しい）

3. **GASとの整合性確認**: GAS側のカウント計算ロジック（`nextCallDate !== todayStr`）と一致することを確認

**修正箇所（1183行目）**:
```typescript
// 変更前
case 'general':
  // 一般カテゴリー（専任他決打合せ <> "完了" + 次電日 <> TODAY() + 状況が一般媒介 + 契約年月 >= 2025/6/23）
  query = query
    .neq('exclusive_other_decision_meeting', '完了')
    .gt('next_call_date', todayJST)  // ← 間違った条件
    .eq('status', '一般媒介')
    .gte('contract_year_month', '2025-06-23');
  break;

// 変更後
case 'general':
  // 一般カテゴリー（専任他決打合せ <> "完了" + 次電日 <> TODAY() + 状況が一般媒介 + 契約年月 >= 2025/6/23）
  query = query
    .neq('exclusive_other_decision_meeting', '完了')
    .neq('next_call_date', todayJST)  // ← 正しい条件
    .eq('status', '一般媒介')
```

## Testing Strategy

### Validation Approach

テスト戦略は2段階のアプローチに従います：まず、修正前のコードでバグを実証する反例を表面化させ、次に修正が正しく動作し、既存の動作を保持することを検証します。

### Exploratory Bug Condition Checking

**Goal**: 修正を実装する前に、修正前のコードでバグを実証する反例を表面化させます。根本原因分析を確認または反証します。反証した場合は、再仮説を立てる必要があります。

**Test Plan**: 次電日が今日より前の一般媒介売主をテストデータとして作成し、サイドバーの「一般」カテゴリをクリックしたときにフィルタリング結果に含まれないことを確認します。修正前のコードでこのテストを実行し、失敗を観察して根本原因を理解します。

**Test Cases**:
1. **次電日が今日より前のテスト**: 次電日が「2026-03-25」（今日より前）の一般媒介売主を作成し、サイドバーの「一般」カテゴリをクリック（修正前のコードで失敗）
2. **次電日が今日より後のテスト**: 次電日が「2026-03-27」（今日より後）の一般媒介売主を作成し、サイドバーの「一般」カテゴリをクリック（修正前のコードで成功）
3. **次電日が今日のテスト**: 次電日が「2026-03-26」（今日）の一般媒介売主を作成し、サイドバーの「一般」カテゴリをクリック（修正前のコードで失敗）
4. **次電日がnullのテスト**: 次電日が `null` の一般媒介売主を作成し、サイドバーの「一般」カテゴリをクリック（修正前のコードで失敗）

**Expected Counterexamples**:

### Fix Checking

**Goal**: バグ条件が成立する全ての入力に対して、修正後の関数が期待される動作を生成することを検証します。

**Pseudocode:**
```
FOR ALL input WHERE isBugCondition(input) DO
  result := listSellers_fixed(input)
  ASSERT expectedBehavior(result)
END FOR
```

### Preservation Checking

**Goal**: バグ条件が成立しない全ての入力に対して、修正後の関数が元の関数と同じ結果を生成することを検証します。

**Pseudocode:**
```
FOR ALL input WHERE NOT isBugCondition(input) DO
  ASSERT listSellers_original(input) = listSellers_fixed(input)
END FOR
```

**Testing Approach**: プロパティベーステストは、保存チェックに推奨されます。理由は以下の通りです：
- 入力ドメイン全体で多くのテストケースを自動的に生成します
- 手動ユニットテストが見逃す可能性のあるエッジケースをキャッチします
- バグのない入力に対して動作が変更されていないという強力な保証を提供します

**Test Plan**: 修正前のコードで他のサイドバーカテゴリ（専任、訪問後他決など）の動作を観察し、その動作をキャプチャするプロパティベーステストを作成します。
