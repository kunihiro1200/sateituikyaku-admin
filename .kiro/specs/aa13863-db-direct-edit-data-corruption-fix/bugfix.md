# バグ修正要件ドキュメント

## はじめに

売主AA13863において、Supabaseダッシュボード等でDBを直接操作した際に複数のデータ破損が発生した。具体的には、`address`フィールドが暗号化文字列のまま表示される、`email`がDBで空欄になりスプシにも空欄が同期された、`first_call_person`がスプシには値があるのにDBが空になるという3種類の破損が確認された。

ユーザーの要求は「DBを直接触った場合のタイプの不具合の対策」であり、DB直接操作によるデータ破損を防ぐ仕組み、または破損が発生した場合の検出・修復の仕組みが必要である。

## バグ分析

### 現在の動作（不具合）

**バグ1: addressフィールドの暗号化問題**

`address`（依頼者住所）はステアリングルールで暗号化対象外と定義されているが、`SellerService.createSeller()`では`encrypt(data.address)`が呼ばれており、DBには暗号化文字列が保存されている。DBを直接操作して`address`を平文で上書きすると、次回のスプシ→DB同期（GAS `syncSellerList`）で再び暗号化されずに保存され、`decryptSeller()`が暗号化文字列として復号しようとして失敗し、暗号化文字列がそのまま表示される。

1.1 WHEN DBを直接操作して`address`フィールドを平文で上書きした後にスプシ→DB同期が実行される THEN システムは`address`を暗号化せずにDBに保存し、フロントエンドで暗号化文字列（`E7eW81t6NSeUskW7it4WYImXWAKFt5mm5PBCbNhNeb5y/FqxDrCTn...`のような文字列）がそのまま表示される

1.2 WHEN DBを直接操作して`email`フィールドを空欄にする THEN システムはその空欄をスプシに同期し、スプシの`メールアドレス`列も空欄になる

1.3 WHEN DBを直接操作して`first_call_person`フィールドを削除・空欄にする THEN システムはスプシの「一番TEL」列の値を無視してDBの空欄状態を維持し、スプシとDBの間で不整合が発生する

### 期待される動作（正しい動作）

2.1 WHEN DBを直接操作して`address`フィールドを平文で上書きした後にスプシ→DB同期が実行される THEN システムはスプシの`依頼者住所`列の値を正しくDBに反映し、フロントエンドで正しい住所が表示される

2.2 WHEN DBを直接操作して`email`フィールドを空欄にした後にスプシ→DB同期が実行される THEN システムはスプシの`メールアドレス`列に値がある場合、その値を暗号化してDBに復元し、スプシへの空欄同期を防ぐ

2.3 WHEN DBを直接操作して`first_call_person`フィールドを削除・空欄にした後にスプシ→DB同期が実行される THEN システムはスプシの「一番TEL」列の値を検出してDBに反映し、スプシとDBの整合性を回復する

### 変更してはいけない動作（リグレッション防止）

3.1 WHEN アプリケーション経由（フロントエンドUI）で売主データを更新する THEN システムは引き続き正常に暗号化・復号化を行い、データを正しく保存・表示する

3.2 WHEN GASの`syncSellerList`（10分ごと）が正常なDBデータに対して実行される THEN システムは既存の正常なデータを上書きせず、変更がない場合はDBを変更しない

3.3 WHEN `name`・`phone_number`・`email`フィールドをアプリケーション経由で更新する THEN システムは引き続きこれらのフィールドを暗号化してDBに保存し、復号化して表示する

3.4 WHEN スプシの「一番TEL」列に値がある売主に対してDB→スプシ同期が実行される THEN システムは`first_call_person`の値を正しくスプシに反映する

3.5 WHEN DBを直接操作せず通常のアプリケーション操作のみを行う THEN システムの全ての既存機能が正常に動作し続ける
