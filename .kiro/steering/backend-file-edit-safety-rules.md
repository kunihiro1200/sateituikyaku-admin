# バックエンドファイル編集安全ルール

## ⚠️ 最重要：バックエンドのビルド失敗は即座に認証エラー・ログインエラーを引き起こす

**バックエンドのビルドが失敗すると、管理画面へのログインが完全に不可能になる。**
**これはビジネスに直接影響する最大級の障害。**

---

## 🚨 2026年5月14日の事故

### 何が起きたか

- `str_replace` ツールで `sitemap.ts` を編集した
- `export default router;` の後ろに古いコードの残骸が残った
- TypeScriptコンパイルエラー → Vercelビルド失敗 → バックエンド全停止
- 管理画面へのログインが完全に不可能になった（認証エラー）

### 根本原因

`str_replace` ツールは「oldStr → newStr」の置換だが、**oldStrの範囲指定が不完全だと古いコードが残る**。
特にファイルの末尾付近の編集で発生しやすい。

---

## ✅ バックエンドファイル編集の必須ルール

### ルール1：大きな変更は `fs_write` で完全上書きする

```
❌ 危険：str_replace で大きなブロックを置換する
✅ 安全：fs_write でファイル全体を書き直す
```

**判断基準**：
- 50行以上の変更 → `fs_write` で完全上書き
- ファイルの末尾付近の変更 → `fs_write` で完全上書き
- 関数全体の置き換え → `fs_write` で完全上書き
- 小さな1〜2行の変更 → `str_replace` でOK

### ルール2：編集後は必ずTypeScriptコンパイルチェックを実行する

```bash
# バックエンドのコンパイルエラーを確認（必須）
cd C:\Users\kunih\sateituikyaku-admin\backend
npx tsc --noEmit
```

**エラーが出たら絶対にプッシュしない。**

### ルール3：コンパイルエラーが出たら即座に修正してからプッシュする

```bash
# エラー確認
npx tsc --noEmit

# エラーがゼロになってからプッシュ
git push origin main
```

## 📋 バックエンド編集時のチェックリスト

- [ ] 変更が50行以上 or ファイル末尾付近 → `fs_write` で完全上書きしたか？
- [ ] `npx tsc --noEmit` でエラーゼロを確認したか？
- [ ] エラーゼロを確認してからプッシュしたか？

---

## 🔴 絶対にやってはいけないこと

```
❌ コンパイルエラーを確認せずにプッシュする
❌ str_replace で大きなブロックを置換してそのままプッシュする
❌ 「たぶん大丈夫」でプッシュする
```

## 🛠️ 万が一ビルドが失敗した場合の緊急対応

### ステップ1：直前のコミットに戻す

```bash
git revert HEAD
git push origin main
```

### ステップ2：Vercelの再デプロイを待つ（1〜2分）

### ステップ3：ログインできることを確認

```
https://sateituikyaku-admin-frontend.vercel.app/login
```

**バックエンド編集後は必ず `npx tsc --noEmit` でエラーゼロを確認してからプッシュする。**

**大きな変更は `str_replace` ではなく `fs_write` で完全上書きする。**
