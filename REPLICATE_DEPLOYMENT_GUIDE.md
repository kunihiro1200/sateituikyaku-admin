# Replicate API 画像処理機能 デプロイガイド

## 📋 実装完了内容

### 1. ✅ Replicate画像処理モジュール作成
- **ファイル**: `replicate_image_processor.py`
- **機能**:
  - 元の部屋の雰囲気を保持（`prompt_strength: 0.35`）
  - シンプルな家具を1〜2個配置
  - 別角度・別の照明で撮影したように見せる
  - 各画像から4パターンのバリエーションを生成

### 2. ✅ スクレイピングサーバーに統合
- **ファイル**: `scrape_server.py`
- **変更内容**:
  - `save_to_supabase()`関数に`process_images`パラメータを追加
  - `process_images=True`の場合、Replicate APIで画像処理を実行
  - 処理済み画像をDBに保存

### 3. ✅ バックエンドAPI対応
- **ファイル**: `backend/src/routes/tateuriPreview.ts`
- **変更内容**:
  - `/api/tateuri/scrape`エンドポイントに`processImages`パラメータを追加
  - スクレイピングサーバーに`process_images`を渡す

### 4. ✅ フロントエンド実装
- **ファイル**: `frontend/frontend/src/pages/TateuriManagePage.tsx`
- **変更内容**:
  - 「画像加工あり」と「画像加工なし」の2つの追加フォームを実装
  - 画像加工ありの場合、`processImages: true`をAPIに送信

## 🚀 デプロイ手順

### ステップ1: Railwayに環境変数を追加

1. **Railwayダッシュボードにアクセス**
   - URL: https://railway.app/
   - プロジェクト: `sateituikyaku-scrape-server`

2. **環境変数を追加**
   - 左メニューから「Variables」をクリック
   - 「New Variable」をクリック
   - 以下を追加:
     ```
     REPLICATE_API_TOKEN=<your-replicate-api-token>
     ```
   - **注意**: トークンはReplicate.comのアカウントページから取得してください

3. **保存**
   - 「Add」または「Save」をクリック

### ステップ2: スクレイピングサーバーをデプロイ

1. **ローカルでコミット**
   ```bash
   cd C:\Users\kunih\sateituikyaku-scrape-server
   git add replicate_image_processor.py scrape_server.py
   git commit -m "feat: Add Replicate API image processing with simple furniture"
   git push origin main
   ```

2. **Railwayが自動デプロイ**
   - Railwayは`main`ブランチへのpushを検知して自動デプロイします
   - デプロイ状況はRailwayダッシュボードで確認できます

3. **デプロイ完了を確認**
   - Railwayダッシュボードで「Deployed」と表示されることを確認
   - ログに`🚀 スクレイピングAPIサーバー起動中...`が表示されることを確認

### ステップ3: バックエンドをデプロイ（既にデプロイ済みの場合はスキップ）

バックエンドは既に`processImages`パラメータに対応しているため、再デプロイは不要です。

### ステップ4: フロントエンドをデプロイ（既にデプロイ済みの場合はスキップ）

フロントエンドは既に画像加工オプションに対応しているため、再デプロイは不要です。

## 🧪 テスト手順

### 1. 管理画面にアクセス
- URL: https://sateituikyaku-admin-frontend.vercel.app/tateuri-manage

### 2. 「画像加工あり」フォームでテスト
1. athomeの物件URLを入力（例: `https://www.athome.co.jp/kodate/...`）
2. 「追加」ボタンをクリック
3. 「取得中...」と表示される（画像処理に時間がかかります）
4. 「✅ 〇〇を追加しました（画像加工済み）」と表示される

### 3. 物件詳細を確認
1. 追加された物件の「確認」ボタンをクリック
2. 画像が元画像 + 4パターンのバリエーション（合計5枚×元の画像数）で表示されることを確認
3. バリエーション画像が以下の特徴を持つことを確認:
   - 元の部屋の雰囲気が保たれている
   - シンプルな家具が1〜2個配置されている
   - 別角度・別の照明で撮影したように見える

## 🐛 トラブルシューティング

### エラー: `REPLICATE_API_TOKEN環境変数が設定されていません`

**原因**: Railwayの環境変数が設定されていない

**解決方法**:
1. Railwayダッシュボードで環境変数を確認
2. `REPLICATE_API_TOKEN`が正しく設定されているか確認
3. Railwayを再デプロイ

### エラー: `画像処理エラー: ...`

**原因**: Replicate APIの呼び出しに失敗

**解決方法**:
1. Replicate APIトークンが有効か確認（https://replicate.com/account/api-tokens）
2. Replicateアカウントに十分なクレジットがあるか確認
3. 画像URLが正しいか確認（athomeの画像URLは有効期限がある場合があります）

### 画像処理が遅い

**原因**: Replicate APIは画像生成に時間がかかります

**解決方法**:
- 画像数が多い場合は、処理に時間がかかることを理解してください
- 最初は少数の画像でテストしてください
- 将来的には、バックグラウンドジョブで処理することを検討してください

## 📝 今後の改善案

### 1. バックグラウンドジョブ化
- 画像処理を非同期で実行
- ユーザーは待たずに他の作業ができる
- 処理完了後に通知

### 2. キャッシュ機能
- 同じ画像URLの場合、再処理せずにキャッシュから取得
- コスト削減

### 3. バリエーション数の調整
- 現在は4パターン固定
- ユーザーが選択できるようにする（1〜10パターン）

### 4. プロンプトのカスタマイズ
- 家具のスタイルを選択できるようにする（モダン、和風、北欧風など）
- 照明の種類を選択できるようにする（自然光、夕方、夜など）
